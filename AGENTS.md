# Doon Repository Workflow

이 저장소의 코드와 문서 작업은 아래 흐름을 기본으로 한다.

## 기본 작업 흐름

1. 작업 목적과 완료 조건을 담은 GitHub Issue를 생성하거나 기존 Issue를 확인한다.
2. 최신 `main`을 기준으로 Issue 전용 브랜치를 만든다.
3. 브랜치에서 필요한 변경과 검증을 완료한다.
4. 변경 파일, 핵심 변경 내용, 검증 결과, 남은 위험을 사용자에게 보고한다.
5. 사용자가 승인하기 전에는 커밋하거나 push하지 않는다.
6. 승인 후 관련 변경만 커밋하고 원격 브랜치에 push한다.
7. Issue를 연결한 Pull Request를 생성한다.
8. PR 검증과 리뷰가 끝난 뒤 사용자 승인을 받고 `main`에 머지한다.

직접 `main`에서 기능을 개발하거나 바로 push하지 않는다. 예외가 필요하면 작업 전에 사용자와 범위를 합의한다.

## 브랜치 이름

브랜치 이름은 다음 형식을 사용한다.

```text
<issue-number>-<short-work-description>
```

예시:

```text
1-login-screen
12-discord-accessibility-reader
```

- Issue 번호를 맨 앞에 둔다.
- 작업 설명은 짧은 영문 kebab-case로 작성한다.
- 하나의 브랜치는 하나의 Issue를 해결한다.

## 커밋 메시지

커밋 제목은 다음 형식을 사용한다.

```text
<type>(#<issue-number>) :: <한국어 작업 요약>
```

예시:

```text
feat(#1) :: 로그인 화면 구현
fix(#12) :: Discord 창 탐색 오류 수정
docs(#18) :: 기술 스택 문서 정리
```

### 허용 타입

- `feat`: 기능 추가
- `fix`: 버그 수정
- `refactor`: 동작 변경 없는 구조 개선
- `test`: 테스트 추가 또는 수정
- `docs`: 문서 변경
- `style`: UI 스타일 또는 포맷 변경
- `chore`: 설정, 도구, 패키징 등 기타 작업

### 작성 원칙

- 제목은 한국어로 짧고 명확하게 작성한다.
- 기본적으로 별도의 설명 본문을 작성하지 않는다.
- 서로 독립적으로 되돌릴 수 있는 변경은 커밋을 분리한다.
- 저장소 상위 지침에서 Git trailer를 요구하면 설명 본문 없이 필요한 trailer만 최소한으로 추가한다.

## 변경 보고와 승인

커밋 전 보고에는 다음 항목을 포함한다.

- 변경한 파일
- 사용자 관점의 변경 결과
- 실행한 테스트와 검증 결과
- 실행하지 못한 검증 또는 남은 위험
- 제안하는 커밋 단위와 커밋 제목

사용자의 명시적 승인을 받은 뒤에만 `git add`, `git commit`, `git push`를 진행한다. 승인 이후 변경 범위가 달라지면 다시 보고하고 승인을 받는다.

## Pull Request와 머지

- PR 제목은 가능하면 커밋 제목 형식을 따른다.
- PR 본문은 관련 Issue, 변경 요약, 검증 결과만 간단히 작성한다.
- PR은 관련 Issue를 연결하거나 자동 종료하도록 작성한다.
- CI 또는 필수 검증이 실패한 상태에서는 머지하지 않는다.
- `main` 머지는 사용자의 명시적 승인 후 진행한다.
- 강제 push와 히스토리 재작성은 별도 승인 없이는 수행하지 않는다.
